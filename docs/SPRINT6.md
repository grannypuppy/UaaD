## 一、项目整体成果概览

UAAD 是一个面向活动发布、活动报名、订单支付、通知触达、个性化推荐与并发抢票场景的全栈系统。经过多轮 Sprint 迭代后，项目已经从基础需求分析、架构设计、前后端功能开发，推进到并发能力验证、安全测试、系统监控、实机演示和最终验收准备阶段。

最终系统形成了较完整的业务闭环：

1. 商户创建活动，活动可经历草稿、预热、发布、下架等状态。
2. 用户浏览活动列表、活动详情、热门活动和推荐活动。
3. 用户对已发布活动发起报名，系统通过 Redis 库存扣减和 Kafka 异步处理保证高并发下的库存一致性。
4. 报名成功后生成订单，用户可以支付订单或在允许状态下取消报名。
5. 系统通过通知模块向用户反馈报名成功、报名失败、订单过期、活动提醒等事件。
6. 推荐模块根据热度、用户行为和协同过滤结果生成活动推荐。
7. Prometheus 与 Grafana 提供系统运行指标观测，JMeter 与 Go benchmark 支持压力测试和抗压能力展示。
8. Sprint 5 补充安全测试、模糊测试和实机演示任务，为最终验收提供风险闭环。

## 二、已实现的功能

### 2.1 用户与认证模块

已实现功能：

- 用户注册。
- 用户登录。
- JWT 鉴权。
- 角色区分：普通用户与商户。
- 前端登录态保存、请求拦截、401 处理和退出登录。
- 后端受保护接口统一鉴权。
- 商户接口通过角色权限控制限制普通用户访问。

运行效果：

- 未登录访问受保护接口返回 401。
- 普通用户访问商户创建活动、发布活动、预热活动等接口返回 403。
- token 过期、格式错误、签名错误时会被拒绝。

### 2.2 活动管理模块

已实现功能：

- 活动创建。
- 活动列表查询。
- 活动详情查询。
- 商户活动列表。
- 活动编辑。
- 活动发布。
- 活动预热。
- 活动下架与过期活动处理。
- 活动分类、地点、价格、容量、开始时间、结束时间等基础字段管理。

活动状态：

- `DRAFT`：草稿状态。
- `PREHEAT`：预热状态，可用于展示或营销预热，但尚未正式开放报名。
- `PUBLISHED`：发布状态，用户可以报名。
- `OFFLINE`：下架或过期状态，不再作为可报名活动展示。

运行效果：

- 商户可以先创建草稿活动，再将活动设为预热。
- 预热活动可以继续发布为正式活动。
- 发布时系统会将库存写入 Redis，形成抢票链路所需的库存缓存。
- 过期活动可被自动下架逻辑处理。

### 2.3 报名与抢票模块

已实现功能：

- 用户报名活动。
- Redis 原子扣减库存。
- 重复报名幂等控制。
- 库存不足处理。
- Kafka 异步消息投递。
- Worker 消费报名消息并落库。
- 报名状态查询。
- 报名取消。
- 报名成功后生成订单。
- 报名失败后回滚或记录失败原因。

运行效果：

- 高并发报名下不会出现库存为负或明显超卖。
- 同一用户重复报名同一活动会被识别，避免重复扣减。
- Kafka 写入已调整为同步写入，降低“接口返回排队成功但消息未进入 Kafka”导致报名长期卡在 `QUEUING` 的风险。
- Go 并发测试、JMeter 压测和 Grafana 监控可以共同验证该链路。

### 2.4 订单模块

已实现功能：

- 报名成功后生成订单。
- 订单列表查询。
- 订单详情查询。
- 订单支付。
- 订单状态流转。
- 订单过期扫描。
- 订单过期后通知用户并回滚库存。

运行效果：

- 用户只能查看和支付自己的订单。
- 支付成功后订单状态更新。
- 过期订单会被后台逻辑处理，避免长期占用库存。

### 2.5 通知模块

已实现功能：

- 报名成功通知。
- 报名失败通知。
- 订单过期通知。
- 活动提醒通知。
- 通知列表查询。
- 未读通知数量查询。
- 标记通知已读。

运行效果：

- 用户在报名、支付、过期等关键流程后能够收到业务反馈。
- 用户不能读取或修改其他用户的通知。
- 未读数量能支持前端角标或通知入口展示。

### 2.6 推荐模块

已实现功能：

- 匿名用户热门推荐。
- 登录用户冷启动推荐。
- 用户行为足够时的协同过滤推荐。
- 热门活动接口。
- 用户行为上报。
- 活动热度分数计算。
- 推荐结果缓存与刷新。

推荐策略：

- 匿名用户或无行为用户：使用热门排序。
- 行为数量较少的用户：使用偏好分类补足热门活动。
- 行为数量较多的用户：使用基于物品的协同过滤，并结合热门和新鲜活动兜底。

运行效果：

- 首页或推荐页可以展示热门活动与个性化推荐活动。
- 浏览行为会影响活动热度和后续推荐结果。

已实现功能：

- 首页与活动列表。
- 活动详情页。
- 登录与注册。
- 商户活动管理。

## 三、掌握的关键技术

项目后端主要使用 Go 技术栈：

- Go 语言。
- Gin Web 框架。

- 如何设计 RESTful API。
- 如何进行用户认证与角色鉴权。
- 如何使用 Redis Lua 脚本保证库存扣减原子性。
- 如何使用 Kafka 解耦高并发报名请求和后端落库处理。
- 如何通过事务、幂等和状态机保证业务一致性。
- 如何通过 Go 测试覆盖 service、handler、worker 和集成链路。

### 3.2 前端技术

项目前端主要使用 React 技术栈：

- React。
- TypeScript。
- Vite。
- React Router。
- Axios。
- Vitest。
- Tailwind CSS。
- ECharts。
- i18next。

关键学习点：

- 如何组织前端路由和页面结构。
- 如何封装 API 请求与统一错误处理。
- 如何处理登录态、token 和 401 跳转。
- 如何将商户端和用户端操作入口进行角色区分。
- 如何通过 Vitest 测试 API 封装和基础页面能力。
- 如何在验收阶段关闭 Mock，使用真实后端进行联调。

### 3.3 数据与中间件技术

使用到的中间件：

- MySQL：持久化用户、活动、报名、订单、通知、行为、推荐分数等核心数据。
- Redis：缓存活动库存、报名用户集合、库存预热标记等。
- Kafka：承接报名消息，削峰填谷，降低高并发请求对数据库的直接压力。
- Prometheus：采集后端指标。
- Grafana：展示监控面板。

关键学习点：

- 数据库适合持久化事实数据。
- Redis 适合承接高频读写和库存原子扣减。
- Kafka 适合将高峰请求异步化，减少同步链路压力。
- 监控系统需要提前埋点，否则问题发生时很难解释系统行为。

## 四、开发流程与团队协作

### 4.1 整体开发流程

项目采用按 Sprint 推进的迭代方式：

1. 需求分析阶段：整理 SRS，明确用户角色、功能范围和非功能要求。
2. 架构设计阶段：设计前后端架构、数据库结构、API 契约和核心数据流。
3. 基础功能阶段：完成用户、活动、报名、订单、通知等基础模块。
4. 并发能力阶段：引入 Redis、Kafka、库存扣减、异步 worker 和压测脚本。
5. 推荐与体验阶段：补充推荐模块、前端页面和业务闭环体验。
6. 验收准备阶段：补充安全测试、模糊测试、监控、实机演示和压测展示。
7. 总结反思阶段：整理最终成果、缺陷复盘、课程建议、个人心得和配套材料。

### 5.2 李业

在项目中承担的角色：

自己提交代码实现了什么，在项目中的位置：

- 我的代码位于后端 `internal/service`、`internal/worker`、`internal/handler` 三层，是系统抢票主链路的核心实现。
- `enrollment_service.go`：报名下单全流程（前置校验 → Redis 原子扣减 → 落库 QUEUING → 投递 Kafka），以及基于状态机的报名取消。
- `order_service.go`：订单支付（乐观锁状态流转）与订单过期扫描 `ScanExpired`（超时未支付自动关闭，并在 MySQL 与 Redis 双端回补库存、通知用户）。
- `enrollment_worker.go`：消费 Kafka 报名消息，在一个事务里完成"报名转 SUCCESS → 生成 PENDING 订单（15 分钟过期）→ enroll_count 自增"，事务失败时补偿回滚 Redis 并标记 FAILED、通知失败。
- `activity_offline_job.go`：幂等批量扫描，把 `activity_at` 已过且非终态的活动转为 OFFLINE。
- 这几块共同支撑了 SRS 中"零超卖红线"与"最终一致性"两条非功能需求的落地。

遇到的主要挑战：

- 高并发下的防超卖与幂等：判断库存、扣减、防重复报名、取号必须是不可分割的原子操作，否则并发下会超卖或一个用户占两个名额。
- 异步链路的一致性：接口为削峰先返回 QUEUING、真正落库交给 Worker，一旦 Worker 落库事务失败，已扣减的 Redis 库存必须补偿回去。
- 多套状态机交织的竞态：报名、订单、活动三套状态机交织，取消、过期、支付可能并发触及同一条记录。

采取的解决方法：

- Redis Lua 原子扣减：把"SISMEMBER 判重 → GET 库存 → DECRBY 扣减 → SADD 入集合 → INCR 取号"写进一段 Lua，由 Redis 单线程保证原子性，用返回值 `-2/-1/正数` 区分"重复报名/库存售罄/成功取号"，杜绝线程锁与数据库锁表。
- 全链路补偿（Saga 思路）：报名落库失败、Kafka 投递失败、Worker 事务失败、订单过期、报名取消每条异常路径都调用 `Rollback`（INCR 库存 + SREM 集合）把库存补回。
- 乐观锁替代悲观锁：遵循"禁止 `SELECT ... FOR UPDATE`"约定，所有状态流转用带条件的原子更新（如 `WHERE status='QUEUING'`、`enroll_count < max_capacity`），靠 `RowsAffected` 判断是否抢到流转，天然幂等、无锁竞争。
- 逾期下线用幂等 WHERE 批更新，重复执行第二遍自然 0 行更新，可安全周期调度。

最终收获：

- 真正理解了"缓存扛并发、DB 保最终一致、MQ 做削峰"这套高并发抢票架构是如何协作的，而不只是名词。
- 学会了在分布式异步链路里用补偿/回滚而非分布式事务来保证一致性，体会到每条失败路径都要想清楚"库存回没回去"。
- 掌握了用乐观锁 + 原子条件更新实现无锁幂等状态机，明白了"幂等不是加个判断，而是让操作天然可重复"。
- 学会了为并发逻辑写可复现的测试：多 goroutine 验证 stock=1 只有 1 个成功、用可注入时钟绕过定时器测过期。

深度反思：

- 异步系统不能只看接口返回：接口返回"排队成功"不代表消息真的进了 Kafka、真的落了库；"最佳努力"的补偿若也失败，记录仍会卡在 QUEUING，需要对账兜底——可靠性是分层的，没有银弹。
- 状态机要先画清楚再写代码：三套状态机交织时，先穷举合法迁移再编码，比边写边补判断更不容易漏掉竞态。
- 先测试后实现很有价值：并发与过期问题靠肉眼 review 很难发现，是测试把"看起来对"变成"可证明对"。

在项目中承担的角色：

- Sprint 1 后端 2 组成员；Sprint 2 后端二组成员；Sprint 3 第一组成员。
- Sprint 6 弹性设计专题报告负责人之一，重点梳理报名抢票链路中的 Redis 原子扣减、Kafka 异步削峰、幂等状态流转和库存回滚补偿。

自己提交代码实现了什么，在项目中的位置：

- 我的工作主要集中在后端报名、库存和订单闭环，相关代码位于 `backend/internal/service`、`backend/internal/worker`、`backend/internal/repository` 以及 `backend/tests`。
- `stock_lua.go` 抽象了 Redis 库存引擎，通过 Lua 脚本把重复报名判断、库存扣减、报名用户集合写入和排队序号生成放在一个 Redis 原子操作中，支撑高并发报名入口。
- `enrollment_service.go` 负责报名请求的前置校验、Redis 扣减、`QUEUING` 报名记录创建、Kafka 消息投递，以及投递失败时的 Redis 回滚和报名失败标记。
- `enrollment_worker.go` 负责消费 Kafka 报名消息，在 MySQL 事务中将报名从 `QUEUING` 推进到 `SUCCESS`、创建 `PENDING` 订单并增加活动报名计数；事务失败时补偿 Redis 并发送失败通知。
- `order_service.go` 的订单支付和 `ScanExpired` 过期扫描通过条件更新控制状态流转，过期订单关闭后同时回补 MySQL 活动库存和 Redis 实时库存，避免名额被长期占用。
- 配套测试覆盖了 Redis Lua 扣减与回滚、Worker 幂等处理、订单过期回滚、取消与过期并发场景，帮助验证抢票链路在异常路径下不会明显超卖或重复回补。

遇到的主要挑战：

- 抢票链路的入口并发压力集中，若直接依赖数据库查询和更新库存，容易在高并发下出现锁竞争、超卖或响应时间抖动。
- 报名接口采用异步削峰后，请求返回 `QUEUING` 并不代表订单已经创建成功，需要处理 Redis 已扣减但 Kafka 投递失败、Worker 事务失败、用户主动取消、订单过期等多条异常路径。
- 幂等控制不仅要防止同一用户重复报名，还要防止 Worker 重复消费、状态重复流转、取消和过期扫描并发触发时造成重复回滚。
- Redis、Kafka、MySQL 三类组件分别承担实时库存、消息缓冲和最终持久化职责，任何一层失败都可能造成状态不一致，必须明确补偿边界和可观测信号。

采取的解决方法：

- 使用 Redis Lua 脚本作为报名入口的第一道闸门，一次性完成 `SISMEMBER` 判重、`GET` 库存检查、`DECRBY` 扣减、`SADD` 记录用户和 `INCR` 生成排队序号，避免把高并发库存竞争直接打到 MySQL。
- 使用 Kafka 将“接收报名请求”和“创建订单落库”解耦，接口层只负责完成准入和消息投递，Worker 层异步完成报名成功、订单创建和活动计数更新，削平瞬时流量峰值。
- 对所有关键状态流转采用条件更新，例如报名只从 `QUEUING` 更新为 `SUCCESS`，订单只从 `PENDING` 更新为 `PAID` 或 `CLOSED`，并根据 `RowsAffected` 判断是否真正抢到状态流转权。
- 设计补偿回滚路径：报名记录创建失败、消息序列化失败、Kafka 写入失败、Worker 事务失败、用户取消和订单过期都会调用 Redis `Rollback`，将库存加回并从已报名集合移除用户。
- 用测试复现容易被忽略的竞态场景，例如订单过期扫描重复执行、取消与过期并发执行、Worker 收到非 `QUEUING` 报名消息，确保重复调用不会造成重复扣减或重复回补。

最终收获：

- 理解了弹性设计不是单一技术点，而是 Redis、Kafka、MySQL、Worker、补偿逻辑和监控指标共同组成的工程体系。
- 掌握了用缓存承接热点写入、用消息队列削峰、用数据库状态机保证最终一致的典型高并发业务架构。
- 对幂等有了更实际的认识：幂等不是简单判断“是否存在”，而是要让每个状态迁移都有明确前置条件，并允许消息重放、任务重跑和接口重试。
- 意识到测试对并发链路非常重要，尤其是库存、订单和取消这类涉及资源归还的逻辑，必须通过可重复测试确认异常路径。

深度反思：

- 当前系统已经能通过 Redis 与 Kafka 承接一定并发，但弹性能力仍偏“应用内补偿”，还缺少更完整的重试队列、死信队列、断路器和统一降级策略。
- Worker 失败后虽然会尽量回滚和标记失败，但如果补偿本身失败，仍需要后续对账任务扫描 `QUEUING`、Redis 库存和 MySQL 订单之间的差异，形成更稳的兜底机制。
- 报名链路在设计初期应先画清楚库存、报名、订单三套状态机，以及每个失败点对应的补偿动作；先写代码再补边界条件，容易遗漏异常分支。
- 后续如果继续演进，我会优先补充 Kafka 消费重试与死信队列、Redis/MySQL 库存对账、核心接口限流和推荐等非核心接口降级，让系统从“能抗压”进一步变成“故障下可恢复、可观测、可解释”。

遇到的主要挑战：

- 行为埋点写入频繁，同步落库会拖慢主接口，异步写入又要避免丢数据。
- 压测需要大量真实账号，但注册接口有限流，直接批量注册会被拒绝。
- 前端从 Mock 切到真实接口时，分页结构和字段容易和后端契约不一致。

采取的解决方法：

- 通知写入采用 best-effort，失败只记录日志、不影响主流程。
- 行为写入做成同步/异步可配，兼顾接口耗时和数据可靠性。
- 压测脚本顺序注册并在被限流时退避，再把注册限流改为可配置、压测前临时放宽。
- 联调以系统设计文档的接口契约为准逐页核对，并把核对项写进验收清单。

最终收获：

- 理解了通知、行为这类旁路模块的设计取舍，知道如何在不影响主链路的前提下可靠地记录业务事件，也明白了 best-effort 写入与强一致写入分别适用的场景。
- 熟悉了 MySQL、连接池配置和 bgroup 测试在后端服务中的协作方式，体会到统一配置和测试覆盖对系统稳定性的作用。
- 完整掌握了从压测账号生成、JMeter 压测计划编写到报告产出的流程，学会了通过限流配置、参数化数据来让压测更接近真实场景。
- 通过前后端联调，更深入理解了接口契约、分页结构和错误码在前后端协作中的重要性，也提升了定位和排查联调问题的能力。

深度反思：

- 旁路模块要先想清楚是否允许失败，best-effort 和强一致是两种不同的设计。
- 压测不只是跑脚本，更要提前准备数据、放开限流，并区分业务失败和系统失败。
- 把联调契约固化成清单比口头确认更能减少返工。

### 5.8 钟志昊

在项目中承担的角色：

- Sprint 2 后端一组成员；Sprint 3 第二组成员。
- 在后端异步处理链路中引入 Redis 与 Kafka：使用 Redis 承担热点数据缓存与请求削峰，使用 Kafka 解耦订单创建流程，提升高并发场景下的吞吐能力和系统稳定性。
- 编写并完善 JMeter 压力测试脚本，对核心下单链路进行并发压测，验证 Redis + Kafka 改造后的 QPS 提升效果和订单冲突修复结果。
- 接入 Prometheus 指标监控，并配置 Grafana 可视化面板，用于观察接口 QPS、响应时间、系统资源与服务运行状态，为后续性能分析和验收演示提供数据支撑。

遇到的主要挑战：

- 高并发下单场景中，传统同步处理链路容易在短时间内承受大量请求，导致接口响应时间上升、数据库压力增大，并可能出现订单冲突或库存状态不一致等问题。
- 压力测试不仅需要能模拟足够真实的并发请求，还要保证测试数据、订单编号和用户参数之间互不冲突，否则压测结果会被业务错误干扰，难以准确反映系统性能。
- 监控体系搭建初期，需要把后端服务指标、压测过程和可视化面板串联起来，才能从数据上判断优化是否真正有效，而不是只依赖接口是否成功返回。

采取的解决方法：

- 通过 Redis 缓存和缓冲热点请求，减少高频访问对数据库的直接冲击；同时引入 Kafka 将下单请求与后续订单处理解耦，使系统可以按消息队列节奏平滑消费请求。
- 针对压测脚本中的订单冲突问题，调整 JMeter 参数化方式和测试数据生成逻辑，使并发请求能够覆盖真实场景，同时避免因为重复数据导致异常结果。
- 接入 Prometheus 采集服务运行指标，并使用 Grafana 构建可视化面板，将 QPS、响应时间、错误率和资源状态等指标集中展示，辅助定位性能瓶颈和验证优化效果。

最终收获：

- 对 Redis 缓存、Kafka 消息队列和后端异步处理链路有了更系统的理解，认识到高并发优化不仅是提高接口速度，更重要的是控制系统各环节的压力传递。
- 掌握了使用 JMeter 进行接口压测的方法，能够通过压测脚本、并发参数和测试数据设计来验证系统在不同负载下的表现。
- 熟悉了 Prometheus 与 Grafana 在项目监控中的基本使用方式，能够通过监控数据观察系统运行状态，并将性能优化结果以更直观的方式展示出来。

深度反思：

- 本次工作让我意识到，性能优化不能只关注单个技术点的引入，而需要从业务流程、数据一致性、请求峰值、失败重试和可观测性等多个角度整体考虑。
- 在之后的开发中，应该更早地设计压测方案和监控指标，让功能开发、性能验证和问题定位形成闭环，避免等到系统接近验收时才集中暴露性能和稳定性问题。
- 同时也需要加强对异步链路异常处理的关注，例如消息消费失败、重复消费和补偿机制等场景，这些问题在普通功能测试中不一定明显，但会直接影响真实高并发环境下的可靠性。

遇到的主要挑战：

- 登录流程涉及多个模块联动（Axios 拦截器、AuthContext、ProtectedRoute、token 存储），任何一环出错都会导致登录态丢失或无限跳转，调试链路较长。
- 分支合并后多个页面的样式和路由同时出现问题，需要逐页排查冲突原因，而不是简单地覆盖代码。
- Sprint 3 的前端交互闭环需要同时对接报名、订单、通知、行为、推荐等多个后端接口，接口契约和分页结构不完全一致，联调时需要逐个确认。
- Sprint 5 转到安全测试时，需要从前端开发视角切换到攻击者视角，理解越权、注入、模糊测试等安全概念，并用 Go Test 框架编写可自动化运行的测试用例。

采取的解决方法：

- 登录流程按"token 获取 → 存储 → 拦截器注入 → 401 清理 → 跳转"的顺序逐步实现和验证，每一步单独测试通过后再串联下一步。
- 合并冲突采用逐文件对比策略，先恢复页面能渲染，再修复交互和样式细节，避免一次性改动过多导致新问题。
- 联调时以 `docs/SYSTEM_DESIGN.md` 中的接口契约为基准，前端 API 封装层统一处理分页结构和错误码差异，降低页面层的适配成本。
- 安全测试先阅读 OWASP Top 10 和项目已有的鉴权中间件实现，再按"未认证、角色越权、水平越权、注入、信息泄露"分类设计测试矩阵，模糊测试通过构造异常 payload 批量覆盖核心接口。

最终收获：

- 完整经历了从页面搭建、组件封装、接口对接、国际化到测试补充的前端开发全流程，理解了前端工程不只是写页面，还包括状态管理、鉴权流程、错误处理和测试覆盖。
- 通过商户端和用户端两套页面的开发，理解了角色权限如何在前端路由和 UI 层面落地，以及前后端权限校验需要双重保障。
- 通过安全测试实践，学会了从防御者和攻击者两个视角审视系统，理解了认证、鉴权、输入校验和依赖安全在工程中的实际落地方式。
- 掌握了用 Go Test 编写安全授权测试和模糊测试的方法，能够将安全检查纳入自动化测试流程，而不是只依赖手动验证。

深度反思：

- 前端开发容易只关注"页面能不能用"，但登录态管理、token 安全存储、401/403 处理这些看不见的细节，才是决定系统是否可靠的关键。如果一开始没有设计清楚鉴权流程，后期修补的成本远高于前期投入。
- 安全测试不应该留到最后一个 Sprint 才补充。如果在每个模块开发完成时就同步编写基础的权限和输入校验测试，验收前的压力会小很多，发现问题的时机也更早。
- 从前端开发切换到安全测试，最大的收获是视角转换：写代码时想的是"怎么让功能跑通"，做安全测试时想的是"怎么让功能出错"。这种对抗性思维对提升代码质量很有帮助，未来开发时会更主动地考虑异常输入和权限边界。
- 合并冲突和联调问题占用了不少时间，根本原因是前后端接口契约和分支管理规范在早期没有完全统一。后续项目应该在第一个 Sprint 就固定接口格式、分页结构和分支合并流程，减少后期返工。

- 我围绕当前项目实际结构进行分析：后端虽然仍是单体部署，但内部已经按 `handler / service / repository / worker` 分层，并形成了用户认证、活动、报名、订单、通知、推荐、行为埋点等较清晰的业务边界。
- 我重点分析了报名抢票链路中的“准微服务化”特征：用户请求先经过报名接口和 Redis 库存准入，再通过 Kafka 把后续落库和订单创建交给 Worker 异步处理。这说明系统虽然没有拆成多个独立服务，但已经使用消息队列完成了部分服务间解耦。
- 在报告中，我提出了后续可行的拆分方案：优先将 Worker 独立进程化，再逐步拆出推荐服务、通知服务，最后根据需要拆分用户认证、活动、报名和订单服务。
- 我也补充了微服务化的收益与风险：收益包括模块独立部署、按负载扩容、推荐和通知等模块独立演进；风险包括分布式事务、服务调用失败、部署复杂度、监控和链路追踪成本上升。

遇到的主要挑战：

- 最大的挑战是项目当前并没有真正落地完整微服务架构，因此不能把报告写成“已经实现了多个微服务”。我需要在课程要求和项目事实之间找到准确表述：既说明当前没有完全微服务化，也要讲清楚项目中已经具备哪些可以向微服务演进的基础。
- 第二个挑战是如何区分“模块化单体”和“微服务”。项目中已经有清晰模块边界，也有 Kafka 异步 Worker，但这些并不等同于每个模块都可以独立部署、独立扩容、独立维护。
- 第三个挑战是微服务拆分容易写得过度理想化。如果不结合当前业务，直接把用户、活动、订单、推荐全部拆出去，反而会让系统在课程项目阶段变得过度复杂。
- 第四个挑战是服务拆分后的一致性问题。报名、库存、订单、通知之间存在强业务关联，如果拆分不当，会引入分布式事务、消息重复消费、补偿失败等新问题。

采取的解决方法：

- 我先从当前系统的真实模块出发，梳理已有边界，而不是直接套用通用微服务模板。报告中保留“当前实践情况”一节，明确说明系统当前仍以单体后端为主。
- 我把 Kafka Worker 作为微服务化演进的切入点进行分析。因为报名接口和 Worker 已经通过 Kafka 解耦，相比直接拆活动服务或订单服务，先把 Worker 独立出来更符合当前系统现状，也更容易解释收益。
- 我采用分阶段演进路线，而不是一次性拆分全部服务：第一阶段继续模块化单体；第二阶段独立 Worker；第三阶段拆推荐和通知；第四阶段再引入网关、服务发现、链路追踪和集中配置。
- 对拆分收益和风险同时说明，避免只讲微服务优点。特别强调课程项目规模下完全微服务化并不一定划算，合理的结论应该是“当前保持模块化单体更稳，后续可以按业务压力逐步拆分”。

最终收获：

- 我对微服务架构有了更实际的理解：微服务不是简单地把代码拆成多个仓库或多个进程，而是围绕业务边界、独立部署、独立扩容、服务通信和故障隔离进行系统设计。
- 我理解了 Kafka 在系统演进中的价值。它不仅是一个消息队列，也可以成为服务之间解耦的边界，使高并发报名链路从同步处理转向异步削峰。
- 我认识到架构设计必须结合项目规模。对于课程项目来说，完整微服务化可能会引入超过收益的复杂度；但通过清晰分层、模块边界和异步消息机制，可以为未来演进保留空间。
- 我也学会了用“当前实践、潜在拆分、通信机制、收益、风险、推荐路径”的结构去介绍一个技术专题，让报告既不夸大实现，也能体现架构思考。

深度反思：

- 这次负责微服务架构专题让我意识到，架构报告最重要的不是把概念讲满，而是要忠实于项目真实状态。如果项目没有落地完整微服务，就应该明确说明“未实际落地”，再分析为什么没有落地以及未来如何演进。
- 我认为本项目当前选择模块化单体是合理的。因为系统仍处在课程验收阶段，业务规模、部署环境和团队维护能力都有限，如果过早拆成多个服务，反而会让调试、部署和数据一致性处理变得更难。
- 真正值得保留的是微服务思想中的“边界意识”。即使不拆服务，也应该让活动、报名、订单、通知、推荐等模块有清晰职责，避免所有业务逻辑混在一起。这样后续要拆分时才有基础。
- 后续如果继续完善这个项目，我会优先建议把异步 Worker 独立部署，并完善消息重试、死信队列、链路追踪和监控告警；等推荐和通知模块有独立性能或迭代需求时，再考虑进一步拆分服务。

### 6.1 已发现并处理的问题

| 问题 | 表现 | 处理方式 | 反思 |
|---|---|---|---|
| 数据库连接失败 | 后端启动时报 MySQL 连接拒绝或 root 权限错误 | 检查 Docker MySQL、端口映射、本地 3306 占用和 `.env` 配置 | 运行依赖应尽早脚本化和文档化 |
| 前端 pnpm 启动失败 | `packages field missing or empty` | 修复 `frontend/pnpm-workspace.yaml` | 工具链配置也属于验收风险 |

| 活动预热状态原先只有概念 | `PREHEAT` 没有真实设置入口 | 新增后端预热接口和前端商户操作 | 状态机中的每个状态都应有明确进入路径 |
| 安全测试初期覆盖不足 | 验收前才集中补安全与模糊测试 | Sprint 5 补充安全授权测试和模糊测试 | 安全测试应更早进入迭代节奏 |
| 压测链路理解成本高 | JMeter、Grafana、Prometheus、Kafka 指标较多 | 在文档中固定压测命令、指标解释和演示路径 | 抗压能力展示要面向评审可理解，而不只是能跑脚本 |

### 6.2 使用智能体过程中的心得体会

正向作用：

- 智能体能快速读取仓库、定位文件、总结功能和整理文档。
- 对错误日志的解释和排查路径很有帮助，尤其是 Docker、MySQL、Kafka、pnpm 等问题。
- 对跨模块变更有帮助，例如活动预热同时涉及后端 service、handler、routes、前端 API、页面操作和文档。
- 对测试补充有帮助，可以根据验收口径生成安全测试、模糊测试、集成测试和运行命令。
- 对最终材料整理很有帮助，可以把分散的文档、测试、报告和运行说明汇总为提交结构。

暴露的问题：

- 智能体可能根据一般经验给出建议，但项目真实代码不一定一致，因此必须要求其先读仓库。
- 智能体容易生成“看起来完整”的文档，但其中的人名、日期、测试结果、截图证据必须由团队核实。
- 智能体不能代替团队理解业务状态机和系统边界，尤其是报名、库存、订单、支付、取消这些关键链路。
- 智能体生成的代码仍需要测试验证，不能只靠文字解释判断正确。

改进经验：

- 提问时给出当前错误日志、所在目录、运行命令和期望结果。
- 让智能体先查文件再回答，减少幻觉。
- 对每次变更要求给出验证命令。
- 对验收材料要求区分“已实现”“已测试”“已记录风险”和“未来建议”。
- 对个人心得、课程反馈等主观材料，由成员自己补充真实经历，智能体只负责结构化。

### 6.3 团队协作反思

本项目最大收获之一是理解到：系统开发不是把功能分别写完就结束，而是要让功能、测试、运行、监控、文档和演示共同闭环。

反思点：

- 如果没有统一运行指南，系统即使能在某个人电脑上跑，也不代表能在验收现场稳定运行。
- 如果没有测试和监控，高并发报名这类功能很难证明“真的可靠”。
- 如果没有明确分工，后期容易出现大家都以为别人会补文档、补测试、补演示脚本的情况。
- 如果文档没有持续维护，最终总结阶段会花大量时间重新追溯项目事实。
- 如果每个成员只了解自己负责的局部，实机演示时很难应对临时问题。

后续类似项目应在一开始就建立：

- 固定任务看板。
- 每周可运行版本。
- 每个 Sprint 的 DoD。
- 最小测试要求。
- 文档入口和材料命名规范。
- 演示脚本和故障排查清单。

### 7.2 对组织形式的建议

建议课程组织上：

- 每个阶段明确必交材料，例如需求文档、系统设计、测试报告、运行指南、复盘报告。
- 每个 Sprint 增加一次短检查，确认系统是否能从零拉起。
- 让团队尽早确定角色分工，但允许根据实际进展调整。
- 要求每个成员至少完成一次代码讲解或演示讲解，避免少数成员承担全部理解成本。
- 验收前增加一次模拟验收，提前暴露环境、脚本、数据和讲解问题。

### 7.3 对教学方法的建议

建议教学方法上：

- 多给“可复现”的评价标准，例如必须提供运行命令、测试命令、截图或日志。
- 对 AI 辅助开发过程设置记录要求，例如保留关键对话、关键 prompt、智能体生成内容和人工修改说明。
- 鼓励学生使用真实工程工具，如 Docker、CI、测试框架、监控系统，而不仅是完成页面和接口。
- 对复杂系统设计引入阶段性评审，帮助学生尽早发现架构过度设计或实现不足的问题。
- 对最终报告提供模板，但保留团队自由发挥空间。

## 八、五个技术维度总结

### 8.1 压力测试

已完成探索：

### 8.2 微服务架构

当前实践情况：

- 当前系统仍以单体后端为主，没有完全拆分为多个独立微服务。
- 这是符合当前项目阶段的选择：课程项目更需要先保证核心业务闭环、可运行性、测试和验收稳定性，过早拆成多个服务会显著增加部署、联调、事务一致性和故障排查成本。
- 虽然没有完整微服务化，但系统内部已经具备一定的服务化边界，包括用户认证、活动、报名、订单、通知、推荐、行为埋点和 Worker 等模块。
- 后端代码按 `handler / service / repository / worker` 组织，业务职责相对清晰，为后续服务拆分提供了结构基础。
- Kafka Worker 已经体现了异步服务化思路：报名请求与后端落库、订单创建被消息队列解耦。报名接口负责快速准入，Worker 负责异步完成较重的数据库事务。

当前不完全微服务化的原因：

- 项目规模仍适合模块化单体，单体部署更易启动、调试和验收。
- 多服务拆分会带来额外的网关、服务发现、配置中心、日志聚合、链路追踪和部署编排成本。
- 报名、库存、订单、通知之间存在较强一致性关系，过早拆分会引入分布式事务和补偿逻辑，增加出错概率。
- 团队在课程周期内的重点是完成功能闭环、压测、安全测试和演示，而不是构建完整生产级微服务平台。

潜在拆分策略：

| 服务 | 职责 | 数据/通信 |
|---|---|---|
| 用户认证服务 | 注册、登录、JWT、用户资料 | MySQL、HTTP |
| 活动服务 | 活动创建、发布、预热、下架、查询 | MySQL、Redis |
| 报名服务 | 报名请求、库存扣减、幂等控制 | Redis、Kafka、MySQL |
| 订单服务 | 订单生成、支付、过期扫描 | MySQL、消息事件 |
| 通知服务 | 通知创建、未读数、已读状态 | MySQL、消息事件 |
| 推荐服务 | 行为收集、热度计算、推荐生成 | MySQL、缓存 |
| 监控服务 | 指标采集、日志聚合、告警 | Prometheus、Grafana |

服务间通信机制建议：

- 同步查询类场景优先使用 HTTP REST 或 gRPC，例如活动详情、订单详情、用户资料查询。
- 异步事件类场景优先使用 Kafka，例如报名成功、报名失败、订单过期、通知生成、推荐热度更新。
- 对强一致要求高的链路，例如库存扣减和订单创建，需要谨慎拆分，优先通过本地事务、状态机、幂等键和补偿机制保证最终一致性。
- 服务之间应统一 trace id 或 request id，便于在拆分后进行链路追踪和问题定位。

推荐技术路径：

- 第一阶段：保持模块化单体，明确 service、repository、handler 边界。
- 第二阶段：将 Worker 独立进程化，报名 API 与消费处理分离部署。该阶段改动最小，因为当前系统已经通过 Kafka 将两者解耦。
- 第三阶段：优先拆出推荐服务和通知服务。这两个模块对主链路影响较小，失败时可以降级或重试，适合作为第一批独立服务。
- 第四阶段：根据业务压力再考虑拆分用户认证、活动、报名和订单服务，同时引入 API Gateway、服务发现、集中配置、链路追踪和统一日志。
- 第五阶段：如果进入生产化部署，再考虑容器编排、弹性扩缩容、灰度发布和服务级告警。

收益：

- 模块独立部署。
- 不同服务可按负载独立扩容。
- 推荐、通知、订单等模块可独立演进。
- 高并发报名链路可以单独优化。

风险：

- 分布式事务复杂度增加。
- 部署和监控成本提升。
- 本课程项目规模下完全微服务化可能超过实际需要。
- 服务调用链路变长，故障定位难度上升。
- 网络调用失败、消息重复消费、服务版本不兼容等问题需要额外治理。

结论：

- UAAD 当前阶段更适合“模块化单体 + 局部异步解耦”的架构。
- Kafka Worker 已经为后续微服务化提供了一个自然切入点。
- 后续如果继续演进，应遵循“先拆异步、再拆旁路、最后拆核心交易链路”的顺序，避免为了微服务而微服务。

### 8.4 弹性设计

当前已采取的措施：

- Redis 原子扣减库存，降低数据库直接承压。实现位于 `backend/internal/service/stock_lua.go`，Lua 脚本一次性完成重复报名判断、库存检查、扣减、写入已报名集合和生成排队序号。
- Kafka 异步削峰，避免报名请求直接把数据库打满。接口层在 Redis 准入后先创建 `QUEUING` 报名记录并写入 Kafka，Worker 再异步完成订单创建。
- Worker 异步处理报名消息。实现位于 `backend/internal/worker/enrollment_worker.go`，Worker 在事务中推进报名状态、创建订单并更新活动报名计数。
- 报名幂等控制，降低重复请求风险。Redis 已报名集合防止入口重复报名，数据库层通过 `QUEUING`、`PENDING` 等状态条件更新防止重复消费和重复状态流转。
- 订单过期扫描和库存回滚，避免资源长期占用。实现位于 `backend/internal/service/order_service.go`，`ScanExpired` 只关闭仍处于 `PENDING` 的过期订单，并回补 MySQL 与 Redis 库存。
- Kafka 同步写入和超时设置，降低消息丢失导致状态卡住的风险。消息写入失败时会将 Redis 扣减补偿回去并把报名标记为失败。
- 健康检查、Prometheus 指标和 Grafana 面板辅助发现运行问题，重点观察 HTTP 延迟、错误率、Worker 处理量和 Kafka lag。

核心链路说明：

- 用户提交报名后，服务先校验活动状态和报名窗口，再调用 Redis Lua 脚本完成原子准入。如果返回重复报名或库存不足，接口直接返回业务错误，不进入数据库写入和 Kafka 阶段。
- Redis 准入成功后，系统写入 `QUEUING` 报名记录并投递 Kafka 消息。这个阶段把“用户请求是否被接收”和“订单是否最终创建”拆开，从而削平瞬时流量峰值。
- Worker 消费消息后，在一个事务内将报名推进为 `SUCCESS`、创建 15 分钟过期的 `PENDING` 订单，并增加活动报名计数。若事务失败，Worker 会回滚 Redis 库存并把报名标记为 `FAILED`。
- 用户取消或订单过期时，系统通过条件更新抢占状态流转权，只有成功从 `QUEUING`、`SUCCESS` 或 `PENDING` 迁移出的请求才执行库存回补，避免并发取消和过期扫描造成重复回滚。

幂等与补偿设计：

- 入口幂等：Redis `enrolled_set` 记录已经通过准入的用户，重复请求不会再次扣减库存。
- 消费幂等：Worker 只处理当前仍为 `QUEUING` 的报名消息，重复消息或已取消消息会被跳过。
- 订单幂等：支付、过期关闭和取消都依赖 `PENDING` 条件更新，避免同一订单被多条路径重复关闭或支付。
- 库存补偿：`Rollback` 同时执行 Redis 库存加一和已报名集合移除，覆盖 Kafka 投递失败、Worker 事务失败、用户取消和订单过期等路径。
- 最终一致性：Redis 承担实时库存，MySQL 记录最终报名和订单状态；系统通过补偿、状态机和测试验证两端逐步收敛。

相关测试与验证材料：

- `backend/internal/service/stock_lua_test.go` 覆盖 Redis 原子扣减、重复报名和回滚。
- `backend/internal/worker/enrollment_worker_test.go` 覆盖 Worker 成功处理、事务失败回滚和非 `QUEUING` 消息跳过。
- `backend/internal/service/order_service_flow_test.go` 覆盖订单过期关闭、通知和库存回滚。
- `backend/internal/service/race_isolation_test.go` 覆盖取消、支付和过期扫描之间的竞态隔离。
- `backend/tests/sprint3_order_expire_test.go` 和 `backend/tests/sprint3_closed_loop_test.go` 覆盖接近真实依赖的闭环场景。

已有收益：

- 高并发报名链路具备一定抗压能力，热点库存竞争首先落在 Redis 原子脚本上，而不是直接落在 MySQL 行锁或表锁上。
- 库存扣减更加稳定，重复报名和库存不足在入口阶段即可被识别，减少后续无效消息和无效订单。
- 请求处理和落库处理解耦，接口可以快速返回排队状态，Worker 按消费能力逐步完成持久化。
- 异步 Worker 可作为后续独立扩容点，后续可以根据 Kafka lag 或 Worker 耗时指标增加消费者实例。
- 异常路径具备补偿闭环，常见失败场景不会长期占用库存，也不会让订单资源无限期悬挂。

不足：

- 暂未引入断路器。
- 暂未系统性实现重试策略和死信队列。
- 暂未实现服务自动扩容。
- 暂未实现负载均衡部署。
- 暂未实现统一降级策略。
- 暂未实现 Redis 与 MySQL 库存对账任务，极端情况下如果补偿本身失败，仍需要人工或后续任务修复状态漂移。
- 暂未对报名接口引入明确的限流策略，峰值流量仍可能压垮 Redis、Kafka 或应用实例。

未来可引入的弹性策略：

- 对 Kafka 消费失败引入分级重试和死信队列。短暂失败先延迟重试，多次失败后进入死信主题，并提供后台补偿或人工排查入口。
- 对 Redis、Kafka、MySQL 等关键依赖引入超时、重试和熔断。重试需要限制次数并加入退避，避免故障期间放大流量。
- 对推荐、通知未读数等非核心接口提供降级结果，例如推荐接口异常时返回热门活动，通知角标异常时不阻塞核心报名流程。
- 将 Worker 独立部署并允许多实例水平扩展。
- 使用 Nginx 或 API Gateway 做负载均衡。
- 使用容器编排平台实现自动恢复。
- 增加 Redis/MySQL 库存对账任务，定期扫描报名、订单和 Redis 集合差异，对异常 `QUEUING`、`FAILED` 或库存漂移进行修复。
- 增加核心链路限流和排队保护，例如按活动维度限制报名 QPS，超限时返回可解释的排队或稍后重试提示。
