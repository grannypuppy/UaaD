# Sprint 6 缺陷反思与课程建议

> 本文档用于单独提交课程要求中的“缺陷与反思”和“课程建议”两部分内容。
>
> 最后更新：2026-06-15

## 一、缺陷与反思

### 1.1 已发现并处理的问题

| 问题 | 表现 | 处理方式 | 反思 |
|---|---|---|---|
| 数据库连接失败 | 后端启动时报 MySQL 连接拒绝或 root 权限错误 | 检查 Docker MySQL、端口映射、本地 3306 占用和 `.env` 配置 | 运行依赖应尽早脚本化和文档化 |
| 前端 pnpm 启动失败 | `packages field missing or empty` | 修复 `frontend/pnpm-workspace.yaml` | 工具链配置也属于验收风险 |
| 报名可能长期卡在 `QUEUING` | Kafka 异步写入可能导致接口先返回成功但消息未实际投递 | 将 Kafka Writer 调整为同步写入并设置超时 | 异步系统不能只看接口返回，还要关注消息可靠性 |
| 活动预热状态原先只有概念 | `PREHEAT` 没有真实设置入口 | 新增后端预热接口和前端商户操作 | 状态机中的每个状态都应有明确进入路径 |
| 安全测试初期覆盖不足 | 验收前才集中补安全与模糊测试 | Sprint 5 补充安全授权测试和模糊测试 | 安全测试应更早进入迭代节奏 |
| 压测链路理解成本高 | JMeter、Grafana、Prometheus、Kafka 指标较多 | 在文档中固定压测命令、指标解释和演示路径 | 抗压能力展示要面向评审可理解，而不只是能跑脚本 |
| 接口契约口径容易分散 | 后端 handler、前端 axios、JMeter 断言可能对同一业务状态采用不同判断口径 | 以统一响应结构和接口契约为依据，明确 202、200、409、401、404 等状态的业务含义 | 契约不是后端单方面的实现细节，而是前端、测试、压测共同依赖的真理源 |
| Mock 到真实 API 切换暴露问题较集中 | 前期前端依赖 Mock 推进较快，但切到真实后端后集中暴露字段、分页、鉴权和状态机差异 | 在联调阶段逐步关闭 Mock，对照接口文档逐页验证真实请求和响应 | Mock 是开发加速器，但不能替代真实联调，应更早进入真实 API 验证 |
| 异步链路异常路径复杂 | 报名接口返回 `QUEUING` 不代表 Kafka 已可靠投递、Worker 已落库、订单已创建 | 通过 Redis 回滚、Kafka 同步写入、Worker 事务、订单过期回补等机制补偿异常路径 | 异步系统必须关注最终一致性、补偿和对账，不能只看接口是否快速返回 |
| 登录态与权限边界容易被低估 | token 存储、Axios 拦截器、401/403 处理、受保护路由和角色权限相互影响 | 前端统一封装鉴权逻辑，后端通过 JWT、角色中间件和资源归属校验兜底 | 安全边界应前后端共同设计，不能只依赖页面隐藏按钮 |
| 组内任务边界后期才完全清晰 | 后期容易出现“谁负责实现、谁负责测试、谁负责演示、谁负责报告”需要反复确认 | 通过 Sprint 文档记录阶段目标、分组、负责人和验收口径 | 组长需要更早把实现、测试、演示、文档责任边界显式化 |

### 1.2 使用智能体过程中的心得体会

在本项目中，智能体主要用于需求拆解、代码阅读、错误排查、文档整理、测试补充和验收准备。整体来看，智能体显著提升了团队处理复杂信息和生成结构化材料的效率，但也暴露出需要人工把关的问题。

智能体带来的正向作用：

- 能快速阅读仓库、定位文件、总结模块功能，帮助团队在较短时间内理解已有代码结构。
- 对错误日志的解释和排查路径很有帮助，尤其是 Docker、MySQL、Kafka、pnpm 等环境问题。
- 对跨模块变更有帮助，例如活动预热功能同时涉及后端 service、handler、routes、前端 API、页面操作和文档，智能体能辅助梳理影响面。
- 对测试补充有帮助，可以根据验收口径生成安全测试、模糊测试、集成测试和运行命令。
- 对最终材料整理很有帮助，可以把分散的文档、测试、报告和运行说明汇总为更清晰的提交结构。

智能体使用中暴露的问题：

- 智能体可能根据一般经验给出建议，但项目真实代码不一定一致，因此不能在未读取仓库的情况下直接采纳。
- 智能体容易生成“看起来完整”的文档，但其中的人名、日期、测试结果、截图证据必须由团队核实。
- 智能体不能代替团队理解业务状态机和系统边界，尤其是报名、库存、订单、支付、取消这些关键链路。
- 智能体生成的代码仍需要测试验证，不能只靠文字解释判断正确。
- 如果没有明确告诉智能体真实分工和课程要求，它可能会把总结报告误写成新的测试核查或文档治理任务。

经过多轮使用后，我们形成了以下经验：

- 提问时应尽量给出当前错误日志、所在目录、运行命令和期望结果。
- 对代码问题，应要求智能体先查文件再回答，减少幻觉。
- 对每次变更，应要求给出验证命令或说明验证方式。
- 对验收材料，应明确区分“已实现”“已测试”“已记录风险”和“未来建议”。
- 对个人心得、课程反馈等主观材料，应由成员自己补充真实经历，智能体只负责结构化和语言整理。

结合各成员的个人反思，团队对智能体使用形成了更具体的认识：

- 智能体适合快速生成 handler 测试骨架、JMeter 脚本骨架、文档目录和排查清单，但接口状态码、业务错误码、权限规则和状态流转必须回到代码与系统设计文档中确认。
- 智能体能帮助学习 Go、Redis、Kafka、Prometheus、Grafana 等陌生技术，但“先做后学”之后仍要回头理解底层机制，否则容易只会套模板，不会处理异常路径。
- 智能体对文档结构帮助很大，但它并不知道团队真实分工。组长在使用智能体整理总结材料时，必须先提供真实成员职责、Sprint 安排和课程要求，否则容易把报告任务误拆成新的工程任务。
- 对涉及安全测试、模糊测试、鉴权、越权等内容，智能体可以提供测试思路，但测试结果和风险等级必须由团队结合真实运行结果判断。
- 对个人心得这类材料，智能体只能辅助润色，不能替代成员本人反思。真实挑战、真实贡献和真实收获必须由成员自己补充。

### 1.3 团队协作反思

本项目最大的收获之一是理解到：系统开发不是把功能分别写完就结束，而是要让功能、测试、运行、监控、文档和演示共同闭环。

团队协作中暴露出以下问题：

- 文档与代码同步有时滞后，导致后续成员需要重新确认真实实现。
- 早期任务边界不够清晰，容易出现某些模块无人持续跟进。
- 部分功能先实现后补测试，导致验收前补测压力较大。
- 前端、后端、测试和演示之间的运行环境要求没有一开始就统一，后期出现端口、数据库、pnpm workspace、seed 数据等问题。
- 对 Kafka、Redis、Prometheus、Grafana 等中间件的依赖关系，早期没有完全用图或脚本固化，导致新人理解成本较高。
- 智能体生成内容需要人工审核，若直接采用，可能出现路径不一致、命令不适配或材料过度泛化。

从成员个人反思中可以看到，协作问题主要集中在以下几类：

- **契约协作问题：** 报名接口、订单接口、分页结构、错误码和鉴权响应需要前端、后端、测试、压测同时理解。如果缺少统一口径，handler 测试、前端拦截器和 JMeter 断言会各写各的，后期对齐成本很高。
- **Mock 与真实联调问题：** 前端早期使用 Mock 能提高开发速度，但真实 API 接入后，`QUEUING` 排队状态、分页字段、401 跳转、活动状态禁用规则等差异会集中暴露。Mock 不应长期替代真实后端。
- **异步链路协作问题：** Redis、Kafka、Worker、MySQL、通知和订单之间不是简单的顺序调用，而是带有补偿和最终一致性的链路。只负责局部模块的同学如果不了解全链路，容易误判问题位置。
- **测试和压测前置不足：** 多位成员都提到测试不应最后补。单元测试中固化的契约会影响 JMeter 断言，安全测试也应随模块开发逐步补充，而不是验收前集中完成。
- **组长协调压力：** 项目范围、Sprint 分工、进度推进、文档维护和最终总结都需要持续把控。如果前期没有明确责任边界，后期很多收口工作会自然集中到组长身上。

对团队协作的进一步反思：

- 如果没有统一运行指南，系统即使能在某个人电脑上跑，也不代表能在验收现场稳定运行。
- 如果没有测试和监控，高并发报名这类功能很难证明“真的可靠”。
- 如果没有明确分工，后期容易出现大家都以为别人会补文档、补测试、补演示脚本的情况。
- 如果文档没有持续维护，最终总结阶段会花大量时间重新追溯项目事实。
- 如果每个成员只了解自己负责的局部，实机演示时很难应对临时问题。

后续类似项目中，应在一开始就建立：

- 固定任务看板。
- 每周可运行版本。
- 每个 Sprint 的完成标准。
- 最小测试要求。
- 文档入口和材料命名规范。
- 演示脚本和故障排查清单。
- 接口契约变更记录，明确状态码、业务错误码、分页结构和鉴权行为。
- Mock 退出计划，规定从哪一轮 Sprint 开始必须接入真实后端。
- 压测、安全测试和监控指标的最小验收口径，避免最后阶段才补齐。

## 二、课程建议

### 2.1 对课程内容的建议

建议课程内容进一步加强“从需求到可运行系统”的完整工程链路。课程可以提供一个完整样例，展示如何从需求规格说明、系统架构设计、API 契约、数据库设计、前后端实现、测试验证、部署运行、监控观察一步步推进。这样学生不仅能知道每份文档应该写什么，也能理解文档和代码之间如何互相支撑。

建议增加 AI Agent 辅助开发的边界说明。课程可以专门讲解如何有效使用智能体，例如如何让智能体先读取项目上下文、如何避免智能体幻觉、如何审查智能体生成代码、如何记录智能体交互过程、如何区分智能体生成内容和团队真实贡献。这样可以帮助学生把智能体当成工程协作工具，而不是简单的代写工具。

建议增加中间件实践内容，例如 Redis、Kafka、Prometheus、Grafana 在一个小型系统中分别解决什么问题。许多学生知道这些工具的名字，但不一定理解它们在真实系统中的位置。通过课程案例讲清楚“为什么需要缓存”“为什么需要消息队列”“为什么需要监控”，会比单独介绍工具命令更有帮助。

建议增加安全测试基础内容，包括认证、鉴权、越权、XSS、SQL 注入、模糊测试和依赖漏洞扫描。本项目在验收前集中补安全测试，说明安全内容如果太晚介入，会带来较大补测压力。课程可以要求每个核心模块开发完成后同步补充基础安全用例。

建议增加压测结果解释方法。课程不只需要介绍如何运行 JMeter 或 benchmark，还应讲解如何看 P95、P99、错误率、吞吐量、业务失败和系统失败之间的区别，以及如何结合监控图表判断性能瓶颈。

建议增加“接口契约与前后端联调”的专题。通过本项目可以看到，前端页面、后端接口、单元测试、集成测试和 JMeter 压测都会依赖同一套 API 契约。课程可以专门讲解如何定义统一响应结构、业务错误码、分页字段、鉴权失败响应和状态机字段，并要求团队在每轮 Sprint 中维护契约变更记录。

建议增加“异步系统与最终一致性”的基础讲解。本项目使用 Redis 和 Kafka 后，报名链路不再是简单同步请求，而是涉及排队、消息投递、Worker 消费、事务落库、库存回滚、订单过期等步骤。课程可以通过案例讲解为什么异步系统不能只看接口返回，还要考虑补偿、幂等、重复消费和对账。

### 2.2 对课程组织形式的建议

建议每个阶段明确必交材料，例如需求文档、系统设计、接口说明、测试报告、运行指南、复盘报告。这样团队不会在最后阶段才集中补材料，也能让教师更早发现项目方向是否偏离。

建议每个 Sprint 增加一次短检查，确认系统是否能从零拉起。很多问题不是代码本身，而是环境、端口、数据库、依赖服务、Mock 开关、seed 数据等运行问题。如果每轮都检查一次，可以减少最终验收前的环境风险。

建议让团队尽早确定角色分工，但允许根据实际进展调整。项目推进过程中，成员能力和任务难度会不断变化，如果分工完全固定，可能导致部分任务推进过慢；如果完全不固定，又容易责任不清。课程可以鼓励团队记录每轮分工调整原因。

建议要求每个成员至少完成一次代码讲解或演示讲解，避免少数成员承担全部理解成本。这样既能提升团队整体熟悉度，也能让每个成员在最终答辩或验收时更有参与感。

建议验收前增加一次模拟验收，提前暴露环境、脚本、数据和讲解问题。模拟验收可以要求团队现场启动系统、展示核心流程、运行测试或压测，并说明已知风险。这比只检查文档更能反映项目真实完成度。

建议课程组织中明确“组长职责”和“成员职责”的边界。组长可以负责范围控制、进度推进、Sprint 安排和材料收口，但不应默认承担所有文档、测试和演示补救工作。每一轮 Sprint 都应明确实现负责人、测试负责人、联调负责人和报告负责人，减少后期责任模糊。

建议每轮 Sprint 结束时要求提交简短复盘，包括完成了什么、遇到什么问题、下轮要调整什么、文档是否同步更新。这样最终报告不需要临时追溯所有历史，也能让团队更真实地反映项目演进过程。

### 2.3 对教学方法的建议

建议课程多给“可复现”的评价标准，例如必须提供运行命令、测试命令、截图或日志。可复现材料能减少主观描述，让评审者更容易判断系统是否真实可运行。

建议对 AI 辅助开发过程设置记录要求，例如保留关键对话、关键 prompt、智能体生成内容和人工修改说明。这样既能体现智能体在项目中的实际作用，也能避免团队把智能体生成内容直接当作最终成果而缺少理解。

建议鼓励学生使用真实工程工具，如 Docker、测试框架、监控系统、压测工具和依赖扫描工具，而不仅是完成页面和接口。通过真实工具链训练，学生更容易理解工程项目中的运行、测试、监控、安全和部署问题。

建议对复杂系统设计引入阶段性评审，帮助学生尽早发现架构过度设计或实现不足的问题。例如在引入 Redis、Kafka、微服务、推荐系统、监控系统之前，可以先让团队说明为什么需要这些技术、解决什么问题、引入后会带来什么复杂度。

建议对最终报告提供模板，但保留团队自由发挥空间。模板可以帮助学生覆盖必要内容，但也应允许团队根据自己的项目特点补充真实问题、真实取舍和真实反思。最终报告不应只追求格式完整，更应体现项目过程中的实际经验。

建议在教学评价中强调“真实贡献说明”。每个成员除了写自己负责的功能，还应说明代码位于哪个模块、解决了什么问题、如何验证、遇到什么挑战。这样可以避免最终反思流于泛泛而谈，也便于教师理解团队内部分工。

建议要求学生对智能体生成内容进行二次说明。比如标注哪些部分由智能体辅助生成，哪些地方由本人修改、验证或否定。这样既承认智能体在项目中的作用，也能训练学生对 AI 输出进行审查和负责的能力。
